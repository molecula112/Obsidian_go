`RWMutex` - это примитив синхронизации, который позволяет нескольким горутинам одновременно читать данные, но только одной горутине изменять их

Очень часто в системах есть сценарий:
```
много чтений  
мало записей
```
Например:
```
cache  
config  
in-memory map
```
Если использовать обычный `Mutex`:
```
goroutine1 read → Lock  
goroutine2 read → ждёт  
goroutine3 read → ждёт
```
Хотя чтение не мешает чтению
```
если только читают → пусть читают все  
если кто-то пишет → стоп всем
```

---
#### Как решает RWMutex
`RWMutex` разделяет два типа доступа:
```
read lock  
write lock
```
#### Правила:
```
много читателей одновременно - можно  
читатель + писатель - нельзя  
несколько писателей - нельзя
```
---
#### Методы RWMutex
```go
var mu sync.RWMutex

mu.RLock() // lock для чтения  
mu.RUnlock()  
  
mu.Lock() // lock для записи  
mu.Unlock()
```
Пример:
```go
var mu sync.RWMutex  
var data int  
  
func reader() {  
	mu.RLock()  
	fmt.Println(data)  
	mu.RUnlock()  
}  
  
func writer() {  
	mu.Lock()  
	data++  
	mu.Unlock()  
}
```
Поведение:
```
reader1 --> RLock  
reader2 --> RLock  
reader3 --> RLock
```
Все читают одновременно, но если приходит writer:
```
writer --> Lock --> ждёт пока все readers закончат
```
---
#### Когда RWMutex хуже обычного Mutex
Если запись происходит часто:
```
writes ≈ reads
```
`RWMutex` может стать медленнее, потому что:
```
больше логики  
больше atomic операций  
больше управления очередями
```
---
#### Почему "писать с lock, читать без" — плохая идея
Потому что:
`читатель может увидеть данные в момент изменения`

Запомни:
```
Если данные могут изменяться, то даже операции чтения должны быть синхронизированы. Одновременное чтение и запись без синхронизации приводит к data race.
```
