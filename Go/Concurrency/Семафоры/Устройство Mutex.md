В Go `Mutex` устроен очень просто:
```go
type Mutex struct {  
	state int32  --> состояние мьютекса 
	sema  uint32  --> runtime semaphore для блокировки goroutine
}
```

`state` — это битовое поле, где хранятся флаги и счётчик ожидающих
Примерно так:
```
| waiters count | starving | woken | locked |
```
Основные части:
1. locked - mutex захвачен
2. woken - одна горутина уже разбужена
3.  starving - включён режим fairness
4. waiters - количество ожидающих goroutine
Все они упакованы в **один `int32`** для атомарных операций

---
#### Lock(): fast path
Когда вызывается:
```go
mu.Lock()
```

Go сначала пытается очень быстрый путь, это по факту захват мьютекса без всякой сложной логики, буквально одной атомарной инструкцией CPU:
```go
atomic.CompareAndSwapInt32(&mu.state, 0, locked)
```
Если получилось:
```
mutex свободен  
горутина получает lock
```
Это занимает буквально несколько CPU инструкций

#### Lock(): slow path
Если mutex уже захвачен:
1. Увеличивается количество `waiters`  
2. горутина пытается ещё раз захватить mutex  
3. Если не получилось — паркуется

Парковка происходит через runtime:
```go
runtime_Semacquire(&mu.sema)
```
Это означает:
```
горутина засыпает  
scheduler переключает CPU на другую горутину
```
---
#### Unlock()
Когда вызывается:
```go
mu.Unlock()
```
Происходит:
1. Снимается флаг `locked`  
2. Если есть ожидающие горутины - одну из них будят `runtime_Semrelease(&mu.sema)`
---
#### Почему Mutex быстрый
Потому что используется комбинация:
```
atomic operations  
+  
runtime semaphore
```
И большинство `Lock()` проходит через fast path
