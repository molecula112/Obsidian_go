`RWMutex` внутри намного интереснее, чем обычный `Mutex`, потому что ему нужно управлять двумя типами пользователей: читателями и писателями
#### Структура RWMutex
Упрощённо в Go это выглядит так:
```go
type RWMutex struct {  
    w           Mutex
    writerSem   uint32 
    readerSem   uint32  
    readerCount int32  
    readerWait  int32  
}
```

1. `w Mutex` - обычный `Mutex` для писателей, он гарантирует в один момент времени может писать только одна горутина
2. `readerCount` - сколько читателей сейчас держат `RLock`, каждый `RLock()` делает `readerCount++`
3. `readerWait` - используется когда приходит писатель, сколько активных читателей должны дождаться писателя
4. `readerSem` - cемафор для блокировки читателей, когда есть писатель
5. `writerSem` - семафор для блокировки писателей, пока есть читатели

Ключевая идея:
`writer имеет приоритет над новыми readers`
#### Как работает RLock()
Когда горутина вызывает `RLock()` происходит `readCounter++`, если `readCounter >= 0` значит писатель не ждет, чтение разрешено
#### Когда приходит писатель
Когда вызывается:
`mu.Lock()` писатель захватывает `w Mutex` и блокирует новых читателей `readerCount -= MaxReaders`, число читателей стало отрицательным поэтому когда новый читатель приходит и делает `RLock()` под капотом он делает `atomic.AddInt32(&readerCount, 1)` и получает `< 0`, это знак что `writer есть - reader блокируется`

```
readerCount ≥ 0 --> писатель не ожидает, читатели могут входить  
readerCount < 0 → писатель ожидает, новые читатели блокируются
```

Далее, писатель ждет старых читателей(сколько читателей уже было)
`oldReaders = readerCount + MaxReaders`, каждый читатель по завершению своей работы делает `RUnlock()` то есть уменьшает `readerWait` 

Когда `readerWait == 0` писатель просыпается и получает доступ, делает свои дела и когда заканчивает делает `mu.Unlock()` тем самым возвращая `readerCount += MaxReaders`, а также будит всех читателей
